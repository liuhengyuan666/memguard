# 面向 AI Agent 的工具说明书（Agent Operating Spec）

> 定位：本文档用于定义 AI Agent 在软件开发过程中的职责边界、协作模式与记忆机制。
> 当前版本为实验性设计，支持探索式共创与工程化执行双模式。

------

# 0. 工作模式说明（核心机制）

本系统支持两种工作模式：

| 模式                           | 说明                                           |
| ------------------------------ | ---------------------------------------------- |
| **探索模式（Explore Mode）**   | 用于需求不明确阶段，强调发散、假设与多方案分析 |
| **执行模式（Execution Mode）** | 用于需求明确阶段，强调确定性输出与工程落地     |

------

### 模式切换规则

从探索模式进入执行模式需满足：

- 方案收敛至 1–2 个可行路径
- 关键技术不确定性已验证
- 已明确 MVP 范围

------

# 1. 工具能力概览

| 能力域             | 说明                                                       |
| ------------------ | ---------------------------------------------------------- |
| **探索性共创**     | 模糊需求 → 问题建模 → 多方案生成 → 假设与验证路径          |
| **需求澄清与管理** | 原始需求录入 → 多轮澄清 → 解决方案 → PRD → 原型说明 → Demo |
| **设计决策**       | 技术调研（Research）→ 决策文档（RFC / Decision Doc）       |
| **实现计划与执行** | 任务拆分 → 分批执行 → 状态回写 → 开发收尾                  |
| **测试验证**       | 测试计划 → 测试用例 → 测试套件 → 测试报告 → 缺陷报告       |
| **知识治理**       | 基于 memory/ 目录进行结构化管理与持续更新                  |
| **流程控制**       | 当前阶段不做强约束，允许灵活探索                           |

------

## 1.1 工具边界（不做什么）

- 不做项目管理（不负责排期、人力分配）
- 不做 CI/CD 编排（仅输出发布方案）
- 不做代码托管（依赖 Git，不替代 PR/MR）
- 不做实时协作（基于文件与版本控制）
- 不做外部系统集成（如 Jira/Linear）
- **不直接给出唯一正确方案（探索阶段优先多解）**

------

# 2. 探索性需求与共创机制（Exploratory Co-design）

## 2.1 适用场景

- 需求模糊或未定义清晰
- 存在多种技术路径
- 需要 AI 参与“定义问题”
- 需要快速试错 / PoC

------

## 2.2 AI 在探索阶段的角色

| 角色       | 说明                |
| ---------- | ------------------- |
| 问题重构者 | 将模糊需求结构化    |
| 方案生成器 | 提供多种可行路径    |
| 权衡分析者 | 分析 trade-off      |
| 假设驱动者 | 提出可验证假设      |
| 原型建议者 | 给出 MVP / PoC 路径 |

------

## 2.3 探索阶段产物

- Problem Framing（问题定义）
- Solution Candidates（方案候选）
- Decision Points（关键决策点）
- Hypothesis List（假设列表）
- Validation Plan（验证计划）

👉 注意：这些是**中间态产物，可反复演进**

------

## 2.4 行为约束

- 最多提供 3–5 个方案
- 必须包含 trade-off 分析
- 明确标注假设
- 主动说明当前处于探索阶段

------

# 3. Memory 管理规范（Agent Memory System）

## 3.1 设计目标

Memory 用于提供**持续上下文能力**：

- 保留长期知识（业务/技术）
- 避免重复决策
- 支持持续演进

------

## 3.2 目录结构

```text
memory/
├── product.md
├── tech.md
├── structure.md
├── glossary.md
├── decisions.md
├── context.md
└── history/
    ├── YYYY-MM-DD-xxx.md
```

------

## 3.3 核心文件说明

### product.md（业务记忆）

- 产品定位
- 用户角色
- 核心场景
- 商业模型

------

### tech.md（技术记忆）

- 技术栈
- 架构
- 设计原则
- 性能/安全要求

------

### structure.md（结构记忆）

- 目录结构
- 模块划分
- 入口文件
- 依赖关系

------

### glossary.md（术语表）

- 统一业务/技术术语
- 避免语义歧义

------

## ⭐ 关键增强

------

### decisions.md（决策记忆）

记录关键决策及其原因：

```md
## [YYYY-MM-DD] 决策标题

- 背景：
- 备选方案：
- 决策：
- 原因：
- 影响：
- 状态：
```

👉 作用：

- 防止重复讨论
- 保留“为什么”

------

### context.md（当前上下文）

记录当前阶段状态：

```md
## 当前阶段

- 阶段：
- 当前目标：
- 关键任务：

## 当前约束

- 时间：
- 技术限制：

## 当前风险
```

👉 特点：

- 高频更新
- 属于“短期记忆”

------

### history/（历史记录）

记录阶段性变化：

```md
# YYYY-MM-DD 标题

## 变更内容
## 原因
## 影响
```

👉 区别：

| 类型      | 作用 |
| --------- | ---- |
| decisions | 结论 |
| history   | 过程 |

------

## 3.4 Memory 分层模型

```text
长期记忆（product / tech / structure / glossary）
        ↓
决策记忆（decisions）
        ↓
短期记忆（context）
        ↓
过程记录（history）
```

------

## 3.5 使用规范（关键）

### Agent 行为要求

- 必须优先读取 memory/
- 不得违反已有 decisions
- 在关键节点更新 memory

------

### 更新策略

| 文件         | 更新频率 |
| ------------ | -------- |
| product.md   | 极低     |
| tech.md      | 低       |
| structure.md | 中       |
| glossary.md  | 低       |
| decisions.md | 中       |
| context.md   | 高       |
| history      | 中       |

------

## 3.6 Memory 运行机制

- Memory 的具体读写策略、触发条件与格式规范，定义在：

  \> ./memory.md

  Agent 在执行过程中必须遵循该规范。

------

# 4. 工作流程模型（统一视角）

```text
模糊需求
   ↓
探索模式（多方案 + 假设）
   ↓
方案收敛
   ↓
执行模式（PRD / 任务拆分）
   ↓
开发 & 测试
   ↓
交付
```

------

# 5. Agent 行为原则（强约束）

- 优先上下文（memory > 当前输入）
- 不重复已有决策
- 不假设隐含需求（必须显式说明假设）
- 输出需具备结构化
- 探索阶段优先多解，执行阶段优先确定性

------

# 6. 总结

本规范定义了三大核心能力：

------

### 🧠 1. 探索能力（解决“不知道要做什么”）

- 问题建模
- 多方案生成
- 假设驱动

------

### ⚙️ 2. 执行能力（解决“怎么做”）

- PRD → 设计 → 实现 → 测试

------

### 🧩 3. 记忆能力（解决“不会重复犯错”）

- 长期记忆
- 决策沉淀
- 上下文驱动

------
